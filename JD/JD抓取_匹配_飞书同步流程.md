# JD抓取、简历匹配与飞书同步流程

## 目标

当后续给到新的公司招聘网站时，按固定流程完成以下事情：

1. 从目标公司官网或可信招聘页抓取岗位数据。
2. 先按发布时间和工作年限规则做硬过滤。
3. 使用统一评分脚本对过滤后的岗位逐条打分，输出模型初筛 Top 50。
4. 对 Top 50 做逐条人工复核，选出最适合的前 10 个岗位，并保存为本地 CSV 产物。
5. 将结果同步到飞书多维表格 `offer` 的 `岗位列表` 表中。

这份说明以当前仓库和当前飞书表结构为准。后续如果飞书字段、表名、默认值改变，必须先重新读取 live schema，再继续写入。

## 标准产物

每个公司建议产出四类文件。

### 1. 原始岗位 CSV

保存路径建议：

`JD/<公司名>/<company>_jobs_<类别>.csv`

最低建议字段：

- `岗位名称`
- `事业部名称` 或可拆分出事业部的信息
- `地点`
- `发布日期`
- `链接`
- `JD内容`
- `原始来源`

### 2. Top10 人工筛选 CSV

保存路径建议：

`JD/<公司名>/<company>_jobs_top10_人工筛选.csv`

建议字段：

- `Rank`
- `RecruitPostName`
- `ProductName`
- `BGName`
- `LocationName`
- `LastUpdateTime`
- `PostURL`
- `Responsibility`
- `ManualMatchReason`
- `PotentialGap`

### 3. 过滤后岗位 CSV

保存路径建议：

`JD/<公司名>/<company>_jobs_筛选后.csv`

用途：

- 只保留“近 3 个月内发布”且“非 5 年以上要求”的岗位
- 作为模型评分脚本的直接输入

### 4. 模型评分结果与 Top50 CSV

保存路径建议：

- `JD/<公司名>/<company>_jobs_评分结果.csv`
- `JD/<公司名>/<company>_jobs_top50_模型初筛.csv`

用途：

- `评分结果.csv` 保留全量可评分岗位、评分拆解、模型理由和风险项
- `top50_模型初筛.csv` 作为后续人工逐条复核的输入

腾讯示例：

- 原始 CSV：`JD/腾讯/tencent_jobs_产品经理.csv`
- 人工筛选 Top10：`JD/腾讯/tencent_jobs_产品经理_top10_人工筛选.csv`

## 标准流程

### 步骤 1：确认抓取范围

需要先确认：

- 公司名称
- 招聘来源
  - 优先公司官网
  - 其次官方招聘站
  - 再其次用户明确给出的第三方页
- 岗位范围
  - 例如：产品经理、AI产品经理、策略产品经理
- 当前执行日期
  - 后续所有“近 3 个月”判断都必须基于这次执行的具体日期
  - 例如在 `2026-06-08` 执行时，发布日期下限应为 `2026-03-08`

### 步骤 2：抓取并整理原始 JD

原则：

- 优先抓完整 JD，而不是只抓标题列表。
- 尽量保留原始岗位链接、发布时间、地点、事业部、完整职责描述。
- 如果官网是分页或动态加载，先确认能否通过接口直接拿数据；不行再走浏览器页面采集。

落地要求：

- 将原始岗位保存为 CSV。
- 文件放到 `JD/<公司名>/` 目录下。
- 如果需要，为该公司补一个抓取脚本，放在：
  - `JD/<公司名>/`
  - 或 `scripts/`

### 步骤 3：先做硬过滤

这是固定规则，先过滤，再评分。

#### 3.1 发布时间过滤

- 排除发布时间早于“执行日往前推 3 个自然月”的 JD
- 必须使用明确日期，不要只写“近 3 个月”
- 例如：执行日为 `2026-06-08` 时，保留 `2026-03-08` 及之后发布的岗位

#### 3.2 工作年限过滤

- 排除明确要求 `5年以上`、`5年及以上`、`五年以上`、`五年及以上` 的岗位
- 这一步是硬过滤
- 过滤完成后，剩余岗位里 `0-5 年` 区间都可以继续比较，不再因为 `2年 / 3年 / 4年` 差异直接淘汰

#### 3.3 过滤产物

- 将过滤后的岗位写入 `JD/<公司名>/<company>_jobs_筛选后.csv`
- 后续模型评分和人工复核都以这个文件为准

### 步骤 4：模型初筛打分

先用统一脚本对过滤后的岗位逐条评分。

#### 4.1 当前评分脚本约束

- 使用系统环境变量 `ARK_API_KEY`
- 固定模型：`doubao-seed-2-0-pro-260215`
- 输入上下文必须包含：
  - 完整简历 `resources/jianli.md`
  - 单条岗位的标题、事业部/产品线、地点、发布日期、链接、职责、要求
- 并发数：`5`
- 单条岗位独立评分，不能把所有岗位一起丢给模型排序

#### 4.2 评分标准

模型打分前，必须明确评分口径，避免分数塌缩到一小段区间。

建议总分 `100`，可以拆成以下维度：

1. `业务场景匹配` `0-25`
   - 是否贴近企业 AI、ToB、流程提效、复杂业务协同
2. `平台 / Agent / 工作流匹配` `0-25`
   - 是否贴近 Agent、RAG、工作流、工具调用、中台、权限、评测闭环
3. `产品落地能力匹配` `0-20`
   - 是否需要 0-1 方案设计、跨团队推进、上线复盘、效果迭代
4. `技术理解匹配` `0-15`
   - 是否要求 LLM、RAG、MCP/Skill、Prompt、产品工程能力
5. `领域迁移成本` `0-15`
   - 领域是不是可平移，短板是否可通过迁移补齐

分数锚点建议：

- `90-100`：极强匹配，几乎就是现有经历的直接映射
- `80-89`：强匹配，只有 1-2 个可解释短板
- `70-79`：中高匹配，可投，但需要面试时补关键差距
- `60-69`：部分匹配，不建议优先进 Top10
- `0-59`：弱匹配或方向不合适

#### 4.3 评分校准

第一次跑新公司时，不要直接全量盲信结果。

正确做法：

1. 先抽一批岗位做试评分
2. 看分数是否过于集中
   - 例如大量集中在 `74-78`
3. 或是否过于分散且不符合直觉
   - 例如明显不合适岗位也拿到 `85+`
4. 如有问题，先调整 prompt / 评分说明，再全量跑

#### 4.4 模型产物

- 输出全量 `评分结果.csv`
- 按分数排序取前 `50`，生成 `top50_模型初筛.csv`

### 步骤 5：人工复核 Top50，再定 Top10

这是核心规则。

模型评分只是初筛，不是最终结论。

正确做法：

1. 完整阅读 `resources/jianli.md`
2. 完整阅读 `top50_模型初筛.csv` 中的岗位
3. 逐条判断模型给出的高分是否真的站得住
4. 在 Top50 中重新按人工判断选出最终 Top10

匹配时重点看：

- 是否属于企业 AI、ToB、流程提效、Agent、平台化方向
- 是否和当前履历里的这些经验同构
  - AI 中台 / 统一接入层
  - 权限与组织映射
  - 合同 / 投标审核
  - 商机推送
  - 运维智能体
  - 多场景复用
  - 评测 / 效果闭环 / 监控
- 是否要求从 0 到 1 产品设计、方案落地、跨团队推进
- 是否需要你当前简历里明显能展开的技术理解
  - Agent
  - RAG
  - Tools / Memory
  - Workflow
  - Prompt / Function Call
  - 评测与 badcase 优化

输出要求：

- 选出最适合的前 10 个岗位
- 每个岗位都要写人工匹配原因
- 最好同时写一条 `PotentialGap` / 注意点，说明短板或面试提醒

### 步骤 6：生成 Top10 CSV

将人工筛选结果写入新的 CSV 文件，放回对应公司目录。

要求：

- 这个 CSV 是后续同步飞书的直接输入
- 不要只保留标题和链接
- 必须保留完整 JD 内容和匹配原因

### 步骤 7：同步到飞书 `岗位列表`

同步前，必须重新读取当前 live schema：

- `lark-cli base +table-list`
- `lark-cli base +field-list`
- `lark-cli base +view-get-visible-fields`

不要假设上一次表名、字段名、默认状态还没变。

#### 当前字段映射

Top10 CSV -> 飞书 `岗位列表`

- `公司` <- 固定写公司名，例如 `腾讯`
- `地点` <- `LocationName`
- `状态` <- 默认写 `未投递`
- `岗位名称` <- `RecruitPostName`
- `事业部名称` <- 建议拼成 `BGName / ProductName`
  - 如果没有 `ProductName`，只写 `BGName`
- `JD内容` <- `Responsibility`
- `匹配原因` <- `ManualMatchReason`
  - 如果有 `PotentialGap`，拼到同一字段中
- `发布日期` <- `LastUpdateTime`
- `链接` <- `PostURL`

#### 当前字段内容格式

为了便于在飞书表格里横向浏览，当前字段内容按下面格式写：

- `JD内容`
  - 开头不要再写 `岗位职责：`
  - 直接从职责正文开始
  - 如果需要补充要求，另起一段写 `任职要求：...`
- `匹配原因`
  - 开头不要再写 `匹配原因：`
  - 直接从人工判断正文开始
  - 如果要补 `PotentialGap`，另起一段写 `潜在短板：...`

也就是说：

- 要删除纯展示型前缀
- 保留真正有信息量的正文
- 只有 `任职要求：`、`潜在短板：` 这类分段标签可以保留

#### 状态写入规则

以后新同步的岗位，默认状态都写：

`未投递`

不要再默认写成 `简历评估`。

#### 去重 / Upsert 规则

未来同步时不要无脑追加。

建议用以下组合作为去重键：

- `公司`
- `岗位名称`
- `链接`

处理逻辑：

- 如果不存在：新建记录，状态设为 `未投递`
- 如果已存在：
  - 更新 `地点`
  - 更新 `事业部名称`
  - 更新 `JD内容`
  - 更新 `匹配原因`
  - 更新 `发布日期`
  - 保留已有 `状态`

也就是说：

- `状态` 是用户手动推进面试流程的字段
- 重跑同步时不要覆盖用户已经改过的状态

### 步骤 8：同步后校验

同步完成后，至少做这些检查：

1. 目标表是不是 `岗位列表`
2. 新增 / 更新条数是否等于预期
3. 随机抽查 3-5 条记录
4. 检查以下字段是否落对
   - `公司`
   - `岗位名称`
   - `状态`
   - `链接`
   - `匹配原因`
5. 确认状态默认是 `未投递`

## 操作注意事项

### 1. 飞书写入前，先以 live schema 为准

这个项目里的飞书表结构是会变的。

因此每次同步前都必须：

- 重新查表名
- 重新查字段名
- 重新查默认视图字段顺序

不要沿用旧脚本里的字段名。

### 2. 当前 Windows 环境下，`lark-cli` 需要走非沙箱执行

在当前这台机器上，飞书用户身份依赖 Windows 凭据存储。

现象：

- 沙箱内执行 `lark-cli auth status --verify` 可能显示 `user token missing`
- 但非沙箱执行是正常可用的

因此在 Codex 里做飞书操作时：

- 需要用非沙箱执行 `lark-cli`
- 否则会误判为未登录

### 3. 匹配方式默认是“模型初筛 + 人工定稿”

当前默认流程是：

- 先按发布时间和年限规则做硬过滤
- 再用统一评分脚本做模型初筛
- 最后对 Top50 做人工逐条复核并定 Top10

不要直接把模型分数当最终结论。

### 4. 抓取时优先官方来源

如果用户给的是第三方职位页，也尽量回到：

- 公司官网招聘站
- 官方招聘平台

这样职位信息更完整、发布时间更准确、链接更稳定。

## 推荐执行顺序

后续接到新公司任务时，按下面顺序做：

1. 确认公司和岗位范围
2. 抓取并保存原始 JD CSV 到 `JD/<公司名>/`
3. 按“近 3 个月 + 非 5 年以上要求”生成 `..._筛选后.csv`
4. 阅读 `resources/jianli.md`
5. 用统一评分脚本跑出 `..._评分结果.csv` 和 `..._top50_模型初筛.csv`
6. 对 Top50 做人工逐条复核，输出 `..._top10_人工筛选.csv`
7. 读取飞书 `offer > 岗位列表` 的当前 schema
8. 按字段名做去重后写入飞书，默认状态 `未投递`
9. 抽查写入结果并反馈
