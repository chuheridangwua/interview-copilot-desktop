四类 Agent 产品横评

Hermes Agent、OpenClaw、Claude Code、Codex 的架构区别

这四个名字都在做“让模型替你行动”，但产品重心不一样：Claude Code 和 Codex 是面向软件工程的 coding agent；OpenClaw 是消息入口和设备节点驱动的个人 AI 助手；Hermes Agent 更像可自托管、可研究、可接入多平台的通用 agent harness。下面按入口、核心 loop、上下文、工具扩展、执行边界和安全模型拆开看。

Hermes Agent Nous Research 生态的多入口 agent；CLI、消息网关、工具集、MCP、记忆、cron、子代理和研究批处理是重点。
OpenClaw 自托管个人助手；长期在线 Gateway 连接 WhatsApp、Telegram、Slack、Discord、节点设备和本地/远程能力。
Claude Code Anthropic 的 coding agent；终端、IDE、桌面、Web、Agent SDK、Managed Agents、MCP、skills、hooks。
Codex OpenAI 的 coding agent；App、CLI、IDE、Web/Cloud、AGENTS.md、沙箱审批、MCP、skills、plugins、subagents。
Runtime Map 外圈是产品入口，中心是“模型使用工具完成任务”的 agent loop。

Hermes Agent

CLI / Gateway / ACP / batch 共用 AIAgent 核心；工具注册表、SQLite 状态、上下文压缩、MCP 和多终端后端组成底座。

强项：自托管、多工具、多平台、研究轨迹。

Claude Code

Claude Code harness 管理工具循环；Claude 模型调用内置工具、MCP、skills、hooks、subagents，并可通过 Agent SDK 嵌入。

强项：代码编辑、终端工作流、Claude 原生能力。

OpenClaw

长期在线 Gateway 是中心；聊天渠道、控制客户端、移动/桌面节点经 WebSocket 连接，agent runtime 负责会话与工具执行。

强项：个人事务自动化、消息入口、设备节点。

Codex

Codex agent 读 AGENTS.md 和 config.toml，在沙箱和审批策略下调用 shell、文件、MCP、skills、plugins 与 subagents。

强项：代码改动、审查、可控执行、OpenAI 生态。

架构剖面

每个产品都可以抽象成六层：入口、核心循环、上下文、工具扩展、执行边界、安全控制。区别主要不在“能不能调用工具”，而在默认信任边界和产品入口。

信任边界有危险命令检测、审批工具、沙箱/容器和多终端后端，但整体是自托管宽工具面，安全依赖配置与运行环境。

优劣势

把它们分成两组更容易理解：Claude Code/Codex 是“代码工作台”，Hermes/OpenClaw 是“自托管行动入口”。

Hermes Agent：宽工具面，研究味更重

  • 优势：多 provider、多入口、多 toolset、MCP、cron、memory、batch trajectory、RL/Atropos 等研究能力，适合把 agent 当成可改造平台。
  • 优势：一个 AIAgent 核心服务 CLI、Gateway、ACP、batch 和 API，平台差异尽量留在入口层，架构上比较清晰。
  • 短板：功能面越宽，部署、凭据、工具白名单、容器边界越需要自己管；beta 状态意味着文档和接口变化可能快。
  • 适合：想要本地/自托管、多模型、消息平台、自动化、研究轨迹和技能生态的人。

OpenClaw：聊天入口和设备节点最突出

  • 优势：Gateway-first 架构让它天然适合长期在线个人助手，从 chat app 触发真实世界任务。
  • 优势：clients 和 nodes 都通过 Gateway WS 连接，节点可以把设备能力暴露给 agent，适合“远程主机 + 本地设备”的组合。
  • 短板：它不是默认围绕代码仓库/IDE 设计的，复杂代码任务通常要接 ACP/Codex/Claude 类 agent 或外部工具。
  • 适合：个人运营、邮件/日历/消息流、家庭服务器、Mac mini/VPS 常驻 agent。

Claude Code：Claude 原生 coding loop

  • 优势：终端、IDE、桌面、Web、Agent SDK 和 Managed Agents 形成一套完整 coding agent 产品线。
  • 优势：CLAUDE.md、auto memory、skills、hooks、MCP、subagents、permissions 都围绕软件工程任务设计。
  • 短板：相比 Hermes/OpenClaw，它不是消息网关或个人设备节点平台；通用个人助理需要额外 MCP/集成。
  • 适合：重度 Claude 用户、代码修复/重构/审查/CI 自动化、想用 Agent SDK 做 Claude-powered agent 的团队。

Codex：沙箱审批和 OpenAI 工程生态

  • 优势：AGENTS.md、config.toml、OS sandbox、approval policy、MCP、skills、plugins、subagents 的组合很适合受控代码执行。
  • 优势：App/CLI/IDE/Web/cloud 加 GitHub/Slack/Linear/automation，使它更像软件工程协作环境，而不仅是一个 CLI。
  • 短板：不是 chat-first 个人助手；OpenAI 官方手册和功能矩阵变化快，跨 surface 能力要以当前文档/环境为准。
  • 适合：希望在本地沙箱、云端任务、代码审查、自动修复、团队插件/技能之间切换的 OpenAI 用户。

怎么选

不要按“谁最强”选，按默认架构是否贴合你的信任边界和入口选择。

主要写代码

  • 优先：Claude Code 或 Codex。
  • 选 Claude Code：你偏 Claude 模型、终端/IDE 体验、Agent SDK/Managed Agents。
  • 选 Codex：你偏 OpenAI 生态、沙箱审批、App/CLI/IDE/Web/cloud 协同。

主要做个人助手

  • 优先：OpenClaw。
  • 它的中心不是代码仓库，而是长期在线 Gateway、聊天渠道、节点设备和真实事务。
  • 如果还要研究/多模型/多工具，可以看 Hermes Agent。

想折腾平台

  • 优先：Hermes Agent。
  • 它的工具、gateway、MCP、skills、batch、RL 方向更适合改造成自己的 agent lab。
  • 代价是你要负责部署、安全、凭据和工具边界。

团队可控执行

  • 优先：Codex 或 Claude Code。
  • Codex 的 OS sandbox + approval policy 更明确；Claude Code 的 managed settings、permissions、hooks 和 SDK 也适合治理。
  • OpenClaw/Hermes 更像自托管平台，需要额外运维硬化。
一句话结论

Claude Code 和 Codex 是“让模型在代码库里可控行动”的产品；OpenClaw 是“让模型通过消息和设备节点在你的生活/工作流里长期行动”的产品；Hermes Agent 是“让你自己搭一个多模型、多工具、多入口 agent 平台”的产品。实际选型时，先问入口在哪里、工具边界在哪里、谁来承担安全责任，再问模型能力。
