8. 你怎么把业务逻辑转化成 AI 可读的结构化工作流？
回答逻辑：
业务流程拆解——输入输出定义——任务分层——工具和提示词设计——验证闭环
具体内容：
【流程拆解】 我会先把人工业务流程拆成输入、处理动作、判断规则、输出和异常分支。比如合同评审里，输入是合同、标书、附件和证照，处理动作包括解析、要求抽取、证据检索、风险判断和人工复核。
【输入输出】 然后定义每个节点的输入输出，不让模型自由发挥。比如某个节点只负责提取字段，输出必须是结构化 JSON；某个节点只负责风险判断，必须带证据和置信度。
【任务分层】 再判断哪些适合 Workflow 固定编排，哪些适合 Agent 动态决策，哪些适合 Tool 调用，哪些必须人工确认。
【验证闭环】 最后设计评测样例和 badcase 反馈，验证这个工作流是否真的解决业务问题，而不是只看一次回答效果。

9. 你怎么设计 Agent 工作流？
回答逻辑：
目标定义——任务拆解——工具边界——状态管理——人工兜底
具体内容：
【目标定义】 设计 Agent 工作流前，我会先定义业务目标和成功标准。比如是降低审核时间、提高商机有效率，还是减少运维重复工单。
【任务拆解】 然后把任务拆成可控步骤。以运维智能体为例，可以拆成问题分类、截图识别、SOP 检索、历史经验召回、方案生成、人工确认和经验沉淀。
【工具边界】 每个工具都要有明确边界，比如查知识库、查历史工单、生成图文指引、写入经验库。不能让 Agent 在没有权限和结构约束的情况下随意调用。
【状态管理】 工作流还要管理短期状态和长期记忆，短期状态服务当前会话，长期记忆只沉淀经过人工确认的稳定经验。
【人工兜底】 高风险、低置信度、证据不足的结果必须进入人工确认，这一点在企业和金融场景尤其重要。

10. 你怎么设计 Tool 的 JSON Schema？
回答逻辑：
工具职责单一——入参明确——出参结构化——错误可处理——权限可校验
具体内容：
【职责单一】 我会先保证每个 Tool 职责单一，不做一个万能工具。比如查知识库、查商户数据、生成报告、写入反馈应该拆开。
【入参明确】 入参要明确类型、必填项、枚举值和约束条件，避免模型传模糊参数。比如商户 ID、时间范围、风险类型、数据源、返回条数都应该有清晰定义。
【出参结构化】 出参也要结构化，包含状态、数据、证据来源、错误码和可读说明。这样 Agent 后续节点才能稳定消费结果。
【错误可处理】 工具失败时不能只返回异常堆栈，而要告诉 Agent 是权限不足、参数缺失、数据为空、服务超时还是内部错误。
【权限可校验】 在金融科技场景里，工具调用必须带用户身份和权限上下文，不能把权限控制只写在 prompt 里。

11. 你怎么设计系统提示词和 Prompt 管道？
回答逻辑：
角色边界——任务目标——输出格式——证据约束——多阶段管道
具体内容：
【角色边界】 我会先在系统提示词里明确模型角色和边界，比如它是初审助手、诊断助手、分析助手，而不是最终审批人。
【任务目标】 然后明确当前任务目标，避免一个 prompt 同时承担抽取、判断、总结、生成建议所有事情。
【输出格式】 输出格式要结构化，比如 JSON Schema、风险等级、证据片段、置信度、建议动作，方便后续系统使用。
【证据约束】 高风险结论必须带证据，没有证据就输出不确定或进入人工复核。这个原则是我在合同/投标评审里一直坚持的。
【多阶段管道】 复杂任务我更倾向多阶段 Prompt 管道：先抽取，再检索，再判断，再总结，而不是把全文和所有要求一次性塞给模型。

12. 你怎么做 AI 功能模块的本地 E2E 验证？
回答逻辑：
真实样例——链路跑通——异常场景——指标记录——回归测试
具体内容：
【真实样例】 我会先准备贴近业务的真实样例，而不是只用理想输入。比如合同评审用长文档、图片证照、缺失响应项；运维智能体用真实截图和历史工单。
【链路跑通】 E2E 验证要覆盖从输入、解析、工具调用、模型输出、结构化结果、前端展示到反馈记录的完整链路。
【异常场景】 还要测试异常场景，比如知识库没召回、工具超时、模型输出格式错误、低置信度、用户输入不完整。
【指标记录】 验证时记录响应时间、成功率、输出格式稳定性、人工采纳率、错误类型和成本。
【回归测试】 每次改 prompt、模型、工具或流程，都要跑核心样例和 badcase 回归，避免修了一个问题又引入新的问题。

13. 你怎么在 1 到 2 天内做出一个 AI 产品 MVP0？
回答逻辑：
收窄目标——选核心链路——用现成组件——先本地跑通——再补体验
具体内容：
【收窄目标】 我会先把目标收窄到一个单点闭环，不做大而全系统。比如不是做完整风控平台，而是做一个商户经营数据风险摘要助手。
【核心链路】 然后只选核心链路：输入样例数据，调用一个或两个工具，生成结构化分析结果，前端展示，并支持人工反馈。
【现成组件】 技术上尽量用现成框架和工具，比如 React/Vue 做简单页面，Node 或 Python 做后端，LangChain/LangGraph/Dify/n8n 编排，SQLite 或本地 JSON 存样例数据。
【本地跑通】 第一目标是本地 E2E 跑通，而不是一开始考虑生产级权限、部署和高并发。
【补体验】 跑通后再补日志、错误提示、样例切换、输出复制和简单评测，这样 1-2 天内可以交付一个能演示核心价值的 MVP0。

14. 你理解 Agent、Workflow、Tool 的区别是什么？
回答逻辑：
Workflow 稳定流程——Tool 是能力接口——Agent 是受约束编排器——按场景选择
具体内容：
【Workflow】 Workflow 适合步骤明确、规则稳定的流程，比如文档上传、解析、检索、审核、报告生成。优点是稳定、可控、成本低。
【Tool】 Tool 是具体能力接口，比如查数据库、查知识库、识别图片、生成报告、发送通知。Tool 要有清晰入参、出参、错误码和权限。
【Agent】 Agent 更像受约束的任务编排器，适合目标明确但路径不完全固定的场景。它可以根据当前信息决定下一步调用哪个工具。
【按场景选择】 我不会所有事情都上 Agent。企业场景里，能用 Workflow 固定的就固定，高不确定任务才用 Agent，并且要加工具边界、结构化输出和人工兜底。

15. 你怎么控制 AI Agent 的幻觉？
回答逻辑：
证据绑定——工具约束——结构化输出——低置信度兜底——人工复核
具体内容：
【证据绑定】 高风险结论必须绑定证据，比如原文片段、页码、知识库来源、数据库字段或工具返回结果。没有证据就不能输出确定结论。
【工具约束】 事实类信息尽量通过工具查，不让模型凭记忆编。比如商户经营数据、历史订单、风险记录都应该来自数据库或文件。
【结构化输出】 输出要结构化，包含判断、依据、置信度、风险等级和建议动作。这样系统可以识别低置信度或字段缺失。
【低置信度兜底】 低置信度、证据不足、工具失败的情况不强答，而是进入人工复核或提示需要补充信息。
【人工复核】 在合同评审、金融风控、授信审核这类场景里，AI 定位是初审和辅助分析，不是最终决策人。

16. 金融科技场景里，AI 产品和普通 AI 工具有何不同？
回答逻辑：
风险更高——证据和审计更重要——权限更严格——结果要可解释
具体内容：
【风险更高】 金融科技场景涉及资金、授信、欺诈、坏账和合规，AI 输出不能只追求流畅和智能，必须考虑责任边界。
【证据审计】 每个关键判断最好能追溯证据来源，比如订单数据、流水、平台表现、历史行为、异常记录。否则业务和风控都很难信任。
【权限严格】 数据权限也更严格，不能让模型访问不该访问的商户数据或内部策略。权限控制必须放在系统和工具层，而不是只靠 prompt。
【可解释】 结果要可解释，不能只输出一个黑盒分数。比如为什么一个商户风险高，是订单波动、退款异常、店铺评分下降，还是库存/现金流压力。产品经理要把这些解释设计进产品里。

17. 如果让你为 FundPark 设计一个 AI 商户融资分析助手，你会怎么做第一版？
回答逻辑：
目标用户——输入数据——分析能力——输出结果——验证指标
具体内容：
【目标用户】 第一版我会面向内部业务或风控同事，而不是直接替代授信决策。
【输入数据】 输入可以是商户基础信息、店铺销售趋势、订单数据、退款率、平台评分、广告投入、库存周转、历史融资记录和异常事件。
