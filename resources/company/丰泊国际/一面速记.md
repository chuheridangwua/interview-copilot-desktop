# 丰泊国际 FundPark 一面速记

## 30 秒定位

这家公司不是普通金融公司，而是跨境/电商金融科技公司。

核心业务是：

- 服务高增长电商企业和线上商户。
- 用数据智能和 AI 风控做营运资金贷款。
- 给客户提供市场洞察、资金支持和增长服务。

这个岗位不是传统产品经理，而是超级 AI 产品经理（SAPM）：

- 要懂产品。
- 要懂业务。
- 要懂技术。
- 要能亲手用 Vibe Coding 做 AI Agent Demo。
- 要能把业务逻辑转成 Agent 工作流、Prompt、Tool 和 JSON Schema。

## 你的核心定位

我不是只会写 PRD 的产品，也不是只会调用大模型的 AI 爱好者。

我是做过企业 AI 产品从 0 到 1 的产品候选人，能把业务流程拆成 AI 可执行的工作流，也能用技术工具快速搭 MVP、做本地验证、解决真实落地问题。

## 最重要的项目映射

### 运维智能体 -> Live Demo 首选

最适合演示。

主线：

用户上传故障截图或描述 -> Agent 识别故障类型 -> 检索 SOP 和历史经验 -> 输出客户可执行图文指引 -> 运维确认 -> 有效经验沉淀进长期经验库。

为什么适合：

- 像真正 Agent，不是简单问答。
- 有知识库、图片输入、短期记忆、长期经验库。
- 能讲工作流、工具、结构化输出、人工确认和效果。
- 结果可量化：周末值守从 6-8 人压缩到 2 人。

### 合同/投标评审 -> 金融风控审核

主线：

复杂材料 -> 文档解析 -> 要求抽取 -> 检索证据 -> 多 Agent 核查 -> 风险提示 -> 人工复核。

迁移到 FundPark：

商户融资资料、订单数据、平台流水、异常风险，也可以用类似方法做资料审查、风险识别、证据绑定和人工复核。

关键表达：

AI 不做最终授信决策，AI 做初审排雷和证据整理。

### 商机智能推送 -> 市场洞察和客户增长

主线：

外部高噪声数据 -> 结构化 -> 企业画像 -> 多维评分 -> 机会推荐 -> 业务反馈。

迁移到 FundPark：

电商商户经营数据、市场数据、平台表现和资金需求，也可以做商户画像、增长机会识别、融资建议和风险提示。

### AI 中台 -> SAPM 底层能力

关键词：

- 模型路由
- 权限隔离
- 失败切换
- 用量日志
- 成本监控
- 多场景复用

迁移到 SAPM：

做 Agent 不能只写 prompt，要考虑工具边界、权限校验、日志、异常处理和效果评估。

## 一面高频问题快速答

### 你为什么想来 FundPark？

FundPark 做的是跨境电商金融科技，AI 在这里不是概念，而是直接影响融资效率、风控质量和客户增长。

这个岗位要求 AI 原生产品、Vibe Coding、Agent 工作流和本地 Demo，和我过去做 AI 中台、合同评审、商机推送、运维智能体的经历比较贴合。

我希望从传统企业内部 AI 转型，进入一个更市场化、更数据驱动、更重视 AI 实际落地的环境。

### 你怎么理解 SAPM？

SAPM 不是传统产品经理，也不是纯研发。

它是能同时理解业务、产品和技术的人：

- 能理解客户和业务问题。
- 能设计产品流程。
- 能把业务逻辑转成 AI 工作流、Prompt、Tool 和 JSON Schema。
- 能用 Vibe Coding 亲手做本地 Demo。

### 你为什么适合？

我有三个匹配点：

- 做过多个 AI 产品从 0 到 1，不是只会讲概念。
- 有技术和产品结合能力，能做需求、原型、工作流、工具接入和本地验证。
- 做过 Agent 类产品，尤其运维智能体，能讲清楚知识库、SOP、短期记忆、长期经验库和人工确认。

### 你没有金融风控经验怎么办？

我会承认没有直接做过金融风控，不会硬讲具体授信策略。

但我能快速从业务流程和数据入手，理解申请、资料、初审、补件、风险判断、审批和放款链路。

我能迁移的能力是：把复杂材料和业务规则拆成 AI 可处理的流程，并且做证据绑定、风险分类、人工复核和 badcase 迭代。

## Live Demo 讲解模板

演示时按这个顺序讲：

- 这个产品解决什么业务问题。
- 用户输入是什么。
- Agent 工作流怎么编排。
- 调用了哪些工具或知识库。
- 输出结构是什么。
- 怎么处理低置信度和错误。
- 怎么做人工确认。
- 开发时遇到什么坑。
- 怎么做本地 E2E 验证。

不要只点页面，要边点边讲系统逻辑。

## 一定要讲的关键词

Vibe Coding：

用 Claude Code、Codex 这类工具快速把想法变成本地可运行原型，但关键逻辑仍然要人工审查和测试。

Agent 工作流：

不是让模型自由回答，而是把任务拆成分类、检索、工具调用、结构化输出、人工确认和反馈沉淀。

Tool JSON Schema：

工具要有明确入参、出参、错误码和权限上下文，不能让模型随便调用。

幻觉控制：

事实类结论必须绑定证据，低置信度进入人工复核，工具查不到就不能强答。

金融科技边界：

AI 可以辅助风险识别和资料审核，但不应该直接替代最终授信决策。

## 不会答时的兜底表达

如果问到具体金融风控模型：

这个方向我没有直接负责过，所以具体授信策略我不会硬讲。但我会先拆业务流程、数据来源和风险指标，再从资料完整性检查、经营风险摘要、异常信号提示这类低风险环节做 MVP，逐步积累反馈和评测集。

如果问到现场开发能力：

我不是纯研发，但我具备产品工程能力。我的优势是在 0 到 1 阶段能快速把业务想法拆成页面、接口、数据结构、Agent 流程和本地 Demo，用 AI 编程工具提高实现速度，再通过测试和日志保证结果可验证。

如果问到为什么不是传统产品：

传统产品更多停留在需求和文档，但这个岗位要求 AI 原生开发流。我过去做项目时已经参与到模型、工具、知识库、前后端和测试验证，所以更接近业务产品技术一体化的工作方式。

核心定位：

产品 + 业务 + 技术 + AI Agent 0 到 1 落地。

最强项目：
