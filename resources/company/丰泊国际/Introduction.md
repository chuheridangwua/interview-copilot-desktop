# 丰泊国际 FundPark 一面准备资料

## 一、公司基础信息

公司名称：丰泊国际（FundPark）

成立时间：2016 年

成立地点：香港

公司定位：面向高增长电商企业的金融科技公司，通过数据智能、AI 风控、数字化融资平台和增长服务，帮助中小电商企业获得高效、灵活、安全的营运资金支持。

一句话理解：FundPark 本质上是“跨境/电商金融科技 + AI 风控 + 增长服务”公司，核心客户是线上商户和高增长电商企业，核心能力是用数据和 AI 判断商户经营质量、资金需求和风险水平，再提供营运资金贷款、市场洞察和全链路增长支持。

发展阶段：已进入快速成长阶段（scale-up）。

公司愿景：赋能高增长电商企业释放发展潜力，推动亚洲电商生态健康繁荣。

核心引擎：数据智能与 AI 风控。

核心服务：

- 精准市场洞察
- 高效资金支持
- 全链路增长服务
- 灵活定制化融资方案
- 实时数据支撑
- 安全高效的营运资金贷款服务

## 三、业务理解

### 1. 客户是谁

核心客户是线上商户、高增长电商企业和跨境电商卖家。

这些客户通常有几个痛点：

- 销售增长快，但现金流周转压力大。
- 平台回款、库存采购、广告投放、物流备货之间存在资金错配。
- 传统银行融资流程慢、材料重、授信不灵活。
- 经营数据分散在平台、店铺、物流、广告、支付和财务系统里，很难被传统金融机构快速理解。

### 2. FundPark 解决什么问题

FundPark 的价值不是简单“放贷款”，而是用数据智能和 AI 风控把电商经营数据转化为融资决策依据。

它要解决的问题包括：

- 如何快速判断一个电商商户是否值得授信。
- 如何用实时经营数据动态评估风险。
- 如何在控制坏账风险的同时支持客户增长。
- 如何让客户获得更灵活、更匹配经营节奏的资金。
- 如何从融资服务延伸到市场洞察和增长服务。

### 3. 金融科技产品的核心矛盾

这个方向天然有一个平衡点：

- 客户侧希望额度更高、审批更快、流程更轻、资金更灵活。
- 公司侧需要控制风险、资金成本、欺诈、坏账、合规和运营效率。

所以产品经理不能只追求“体验好”，也不能只追求“风控严”。真正的产品价值是把客户增长和风险控制放在一个可度量、可迭代的系统里。

## 四、岗位信息

岗位名称：超级 AI 产品经理（SAPM）

岗位标签：

- 语义类 AI
- AI 原生开发流
- 打破“不可能三角”
- 精通 Vibe Coding
- 真正的实干家（Doer）
- 一人成团的独立能力
- 认知超能力
- 编排智能体工作流

### 1. 岗位一句话理解

这个岗位不是传统产品经理，也不是纯研发，而是能同时理解产品、业务、技术，并且能亲手用 AI 编程工具快速做出 Agent MVP 的 AI 原生产品经理。

面试官很可能不只听你讲概念，而是要看你能不能：

- 理解复杂业务逻辑。
- 把业务流程转成 AI 可读的结构化工作流。
- 设计 Prompt、Agent、Tool、JSON Schema。
- 用 Vibe Coding 快速做出本地 Demo。
- 解释系统架构、真实问题、踩坑和解决方案。

### 2. 核心要求

打破“不可能三角”：

你要有深厚的产品感觉和业务同理心，同时具备很强的代码实现能力。不要求传统 CS 门槛，但要求能熟练使用大模型辅助工具极快速开发和调试。

真正的实干家：

你要能独立用 Vibe Coding 的方式做 AI Agent 产品，不只是纸上谈兵。

一人成团的独立能力：

你要能独立负责 AI 原生产品从 0 到 1 的生命周期。比如在 1-2 天内完成一个网站或核心功能的本地 Demo 开发。

认知超能力：

要求极强自驱学习能力、严密逻辑和沟通能力。

### 3. 核心职责

- 将业务逻辑和产品需求，转化为 AI 可读的结构化提示词工作流和系统指令。
- 编排智能体工作流。
- 设计 Tool 工具描述和 JSON Schema。
- 设计提示词管道。
- 快速打样，并对新的 AI 功能模块进行日常 E2E 本地环境验证。

### 4. 面试硬性要求

现场演示（Live Demo）：

面试时必须现场投屏演示一个自己亲手做过的 AI Agent 智能体产品。

深度复盘：

需要详细拆解当时的设计逻辑、系统架构、开发过程中的真实瓶颈、踩过的坑以及如何解决。

## 六、项目与岗位映射

### 1. 合同/投标智能评审 -> AI 风控与融资审核

合同/投标评审的核心能力是从复杂材料中抽取关键信息、识别风险、绑定证据、交给人工复核。这和金融科技里的授信审核、资料审查、风险提示有相似逻辑。

可迁移表达：

我做合同/投标智能评审时，不是让 AI 直接替代审批，而是让 AI 做初审排雷，把风险点、原文证据、页码和判断依据输出给人工复核。这个思路可以迁移到 FundPark 的商户融资场景，比如对店铺经营数据、流水、平台订单、库存和异常风险做结构化审查，AI 负责线索识别和证据整理，人负责最终授信决策。

### 2. 商机智能推送 -> 电商市场洞察与客户增长

商机智能推送的核心能力是从外部高噪声数据里识别高价值机会，结合企业画像做评分和推荐。这和 FundPark 提供精准市场洞察、帮助电商客户把握增长机会有相似点。

可迁移表达：

我做商机推送时，核心不是爬数据，而是把外部高噪声信息结构化，再结合企业画像判断哪些线索值得跟进。FundPark 面向电商商户时，也可以用类似逻辑分析市场趋势、店铺经营、平台变化和资金需求，给客户提供增长建议和融资建议。

### 3. 集团 AI 中台 -> SAPM 的 AI 原生产品底层能力

AI 中台体现的是平台治理能力：模型接入、路由、权限、监控、失败切换、用量日志和成本控制。这和 SAPM 要设计 Agent 工作流、Tool、JSON Schema、系统指令非常相关。

可迁移表达：

我做 AI 中台时已经不是单点应用思路，而是在做多场景复用和治理。对于 Agent 产品，我会关注工具边界、权限校验、调用日志、失败兜底和效果评估，而不是只写一个 prompt。

### 4. 运维智能体 -> Agent Live Demo 最适合展示的项目

运维智能体最符合这次岗位的 Live Demo 要求，因为它天然包含 Agent 工作流、工具调用、知识库、图片输入、短期会话、长期经验库和用户反馈。

推荐演示主线：

用户上传截图或描述故障 -> Agent 识别故障类型 -> 检索 SOP/历史经验 -> 输出可执行图文指引 -> 运维人员确认 -> 有效经验沉淀进长期经验库。

面试中可以强调：

这个项目不是简单知识库问答，而是把故障处理流程产品化：输入有截图和文本，过程有知识检索和状态记忆，输出是客户可执行的操作指引，结果还会反哺经验库。

## 七、Live Demo 准备建议

### 1. 优先演示哪个项目

首选：运维智能体。

原因：

- 最像 AI Agent 产品。
- 可以讲清楚工具调用、知识库、短期记忆、长期经验库。
- 可以展示从输入到输出的完整链路。
- 和 JD 的 Agent 工作流、Tool、Vibe Coding、E2E 验证最贴合。

备选：合同/投标智能评审。

原因：

- 业务价值强，风险控制和证据绑定清晰。
- 能体现多 Agent 核查、人机协同和幻觉控制。
- 可以迁移到金融科技风控审核。

### 2. Demo 讲解结构

演示时不要只点页面，要边演示边讲架构：

- 这个产品解决什么业务问题。
- 用户输入是什么。
- Agent 工作流怎么编排。
- 调用了哪些工具或数据源。
- 输出结构是什么。
- 怎么做低置信度和人工兜底。
- 开发过程中遇到过什么坑。
- 如何做本地 E2E 验证。

### 3. 一定要准备的坑

面试官可能会问：

- 这个 Agent 是真工具调用，还是只让模型自由回答？
- Tool 的入参和出参怎么设计？
- JSON Schema 怎么约束输出？
- 低置信度怎么办？
- 知识库召回错了怎么办？
- 如果模型幻觉怎么办？
- 如果本地 Demo 现场失败怎么办？
- 你亲手写了哪些部分？

## 八、面试表达主线

我和这个岗位匹配的点，不是我“知道 AI 很火”，而是我已经做过多个 AI 产品从 0 到 1，能把业务流程拆成 AI 可执行的工作流，也能用技术工具快速做 MVP 和本地验证。

可以这样说：

我理解这个岗位不是传统产品经理，也不是纯开发，而是希望找一个能把业务、产品和技术打通的人。我的经历刚好比较贴合：我做过业务调研和需求拆解，也做过 AI 中台、Agent 工作流、工具接入、知识库和本地 MVP。尤其运维智能体和合同/投标智能评审，都不是只写文档，而是真正跑通了输入、检索、工具调用、结构化输出、人工复核和反馈迭代。
